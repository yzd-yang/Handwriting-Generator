## 一、验证策略总览

### 1.1 四层验证模型

```
┌─────────────────────────────────────────────────────┐
│ L4 上线后监控  │ Sentry 错误率、接口延迟、磁盘占用  │ 上线后 1 周
├─────────────────────────────────────────────────────┤
│ L3 端到端验收  │ 手工冒烟 + 压测 + 浏览器实测       │ 每个阶段结束
├─────────────────────────────────────────────────────┤
│ L2 集成测试    │ 集成测试 + docker compose 联调     │ 每次合并
├─────────────────────────────────────────────────────┤
│ L1 单元测试    │ pytest（后端纯函数）+ vitest（可选）│ 每次提交
└─────────────────────────────────────────────────────┘
```

每一层都要有**明确的通过标准**，缺一层不算完成。

### 1.2 验证环境

| 环境 | 用途 | 数据 |
|------|------|------|
| 本地开发（Windows） | L1/L2 单测与冒烟，**必须验证文件句柄问题**（项目主战场是 Windows） | 真实字体、小样例文本 |
| CI（GitHub Actions，Linux） | L1/L2 自动化，docker build 验证 | 同上 |
| Staging（Docker Compose） | L3 全链路 + 压测 | 完整字体、压测文本 |
| Production | L4 监控 | 真实流量 |

### 1.3 验证原则

1. **每个改动都要有可执行的验证步骤**（命令或操作），不接受"应该没问题"。
2. **回归优先**：先确认旧功能没坏，再验新行为。
3. **负面测试**：不仅要测"正常能跑"，还要测"异常输入不崩"。
4. **Windows 优先**：项目部署在 Windows（有 pandoc msi、文件句柄问题），Linux CI 通过 ≠ Windows 能跑。
5. **留证据**：关键验证截图/日志贴到 PR 描述。

### 2.8 CPU 任务挪线程池（P1）

**验证方法**：并发时事件循环不阻塞。

| 场景 | 操作 | 预期 |
|------|------|------|
| 单任务进度流畅 | 提交大文本任务，WebSocket 监听 | 进度更新间隔 < 1s，不卡顿 |
| 并发任务不影响其他接口 | 2 个大任务渲染中，同时 curl `/api/fonts_info` | fonts_info 响应 < 500ms（之前会被阻塞） |
| 多页渲染进度 | 生成 20 页内容 | progress 从 45→92 平滑推进 |

**验证脚本**：

```python
# 同时跑 2 个大任务 + 持续 ping fonts_info
import threading, time, requests
def gen():
    submit_generate(text="测试\n"*5000, ...)
def ping():
    while True:
        r = requests.get("http://127.0.0.1:5005/api/fonts_info")
        print("fonts_info:", r.elapsed.total_seconds())
        time.sleep(1)

t1 = threading.Thread(target=gen); t2 = threading.Thread(target=gen)
t1.start(); t2.start(); ping()
# fonts_info 响应时间应保持低位，不因渲染任务而飙升
```

**通过标准**：渲染期间 fonts_info P95 < 500ms，进度推送流畅。

### 4.2 性能基线（优化前，来自 README）

| 接口 | 并发 | 平均耗时 | P95 | 失败率 |
|------|------|----------|-----|--------|
| POST /generate | 8 | 9.4s | 17s | 0% |
| GET /fonts_info | 8 | 4.8s | 7.4s | 0% |

> ⚠️ 注意：fonts_info 平均 4.8s **异常高**（本应 <100ms），疑为被渲染任务阻塞事件循环。这是 2.8 要解决的核心症状。

### 4.3 优化后目标

| 接口 | 指标 | 目标 |
|------|------|------|
| GET /fonts_info（渲染中并发请求） | P95 | **< 500ms**（核心改善点） |
| POST /generate | P95 | ≤ 17s（不劣化） |
| 渲染时 WS 进度推送间隔 | - | < 1s（流畅） |
| 首屏 JS chunk | 体积 | 较优化前下降 ≥ 20% |
| 日志文件单文件 | 大小 | ≤ 10MB（轮转） |

**对比方法**：优化前后各跑一次相同参数压测，记录指标到 `doc/re/perf-baseline.md`。

## 六、上线后监控（L4）

> 上线不是终点，持续监控 1 周确认无回归。

### 6.1 Sentry 监控

| 指标 | 关注点 | 告警阈值 |
|------|--------|----------|
| 错误率 | 新增异常 | > 1% 触发告警 |
| identify_distance 相关错误 | 是否还有崩溃 | 应降为 0 |
| PermissionError（文件句柄） | Windows 清理问题 | 应大幅下降 |
| "task failed" 事件 | 错误传递是否正常 | 正常业务失败，非静默 |

### 6.2 接口监控

| 指标 | 工具 | 告警 |
|------|------|------|
| `/api/fonts_info` P95 | 日志/APM | > 1s 告警（优化前 4.8s） |
| 任务完成时间 P95 | 日志统计 | > 30s 告警 |
| 503 queue_full 频率 | 日志统计 | 突增告警 |
| 磁盘占用 | `du -sh backend/temp` | `temp/` 持续增长告警（清理失效） |

### 6.3 日志巡检

上线后每天看一次 `logs/app.log`：
- 有无新增 Traceback（特别是拆分后新模块的 import 错误）
- 有无大量 `remove_dir failed`（清理失效）
- 有无用户文本泄漏（grep 文本特征）

### 6.4 回滚预案

| 触发条件 | 动作 |
|----------|------|
| Sentry 错误率 > 5% | 立即回滚到上一 tag |
| 核心接口不可用 | 回滚 + 查日志 |
| 磁盘被 temp 撑满 | 回滚 + 手动清 temp |
| 大量用户反馈生成失败 | 回滚 + 复现定位 |

回滚命令：
```bash
git tag -l "v-pre-*"            # 找回滚点
git checkout <tag>              # 或 reset --hard
docker compose up -d --build    # 重建
```
