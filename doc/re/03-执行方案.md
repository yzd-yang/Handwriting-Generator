# 03 · 执行方案

> 把 `01-问题分析.md` 的 29 条问题和 `02-修改方案.md` 的改法，落成**可执行的分阶段计划**。
> 包含：阶段划分、任务清单、依赖关系、工时估算、风险与回滚、分支与提交规范。
> 日期：2026-06-20

## 一、总体策略

### 1.1 分阶段原则

| 阶段 | 目标 | 节奏 | 验收门槛 |
|------|------|------|----------|
| **P0 紧急修复** | 堵住安全漏洞与必现崩溃 | 1-2 天 | 不引入新 bug，线上可立即部署 |
| **前端版本核实** | 确认依赖版本真实性 | 0.5-1 天 | `npm install` + `build` 通过 |
| **后端重构** | 拆模块、修健壮性、提性能 | 5-7 天 | 单测通过 + 集成测试通过 |
| **前端重构** | 拆 HomeView、补错误处理 | 4-5 天 | 构建通过 + 手工冒烟通过 |
| **工程化** | CI、Docker、lint、测试 | 3-4 天 | CI 全绿、docker compose up 可用 |
| **收尾优化** | P2/P3 锦上添花 | 按需 | 不阻塞上线 |

**总工期估算**：约 **14-19 人日**（单人全职）。可并行则缩短。

### 1.2 分支与提交规范

- 主分支：`master`（当前默认）
- 每个阶段开一个长生命周期分支，每个任务一个短分支/PR：
  - `fix/p0-security` —— P0 安全修复
  - `fix/p0-identify-crash` —— identify 崩溃
  - `fix/p0-error-propagation` —— 错误传递
  - `refactor/backend-split` —— 后端拆分
  - `refactor/frontend-homeview` —— 前端重构
  - `chore/ci-docker` —— 工程化
- **提交信息**遵循 Conventional Commits（与现有 `release.config.js` 配合）：
  - `fix(security): 移除硬编码 DSN 到环境变量`
  - `refactor(backend): 拆分 app.py 为 routes/services/utils`
  - `feat(ci): 增加 GitHub Actions lint+build 门禁`
  - `chore(deps): 锁定后端依赖版本`
- 每个 PR 必须关联 issue 号（若有）。

### 1.3 回滚原则

- 每个阶段合并前打 git tag（如 `v-pre-refactor-backend`），出问题可 `git reset --hard`。
- P0 修复单独成 PR，可独立 hotfix 上线，不依赖后续重构。
- 后端拆分采用"**先并存后切换**"：新模块就绪、测试通过后，再让 `app.py` 改为 `from routes import ...`，最后删旧代码。每步可独立 revert。

## 三、阶段二：前端依赖版本核实（0.5-1 天）

> 目标：确认 `package.json` 那批异常版本号（vue-router 5.x / vite 8.x / ts 6.x / eslint 10.x / tailwind 4.x）是否真实可装，决定后续前端工作是否阻塞。

### 任务 · 依赖版本核实（对应问题 3.1）

| 项 | 内容 |
|----|------|
| 工时 | 0.5-1 人日 |
| 涉及文件 | `frontendVite/package.json`、`package-lock.json`、`node_modules` |
| 依赖 | 无 |
| 风险 | **高**。若版本是工具篡改，重置后可能有兼容性 bug |
| 回滚 | 恢复 `package.json` + `package-lock.json` 备份 |

**步骤**：
1. 检查 `node_modules` 实际版本：
   ```
   type frontendVite\node_modules\vue-router\package.json | findstr version
   type frontendVite\node_modules\vite\package.json | findstr version
   ```
2. 跑 `npm ls vue-router vite eslint typescript tailwindcss`。
3. **分支判断**：
   - 若 `node_modules` 装的是真实稳定版（如 vue-router 4.x）而 `package.json` 写 5.x → 说明 lock 文件救了它，但 `package.json` 版本号是错的，需修正 `package.json`。
   - 若 `node_modules` 装的也是 5.x/8.x 这类异常版本 → 必须全量重置（删 `node_modules` + `package-lock.json`，改 `package.json` 为真实稳定版，`npm install`）。
4. 重置后跑 `npm run build`，逐个解决兼容性问题：
   - vue-router 5→4：`createRouter`/`createWebHistory` API 兼容，主要是类型变化。
   - tailwind 4→3：`tailwind.config.js` 写法可能要调（v4 配置方式变了）。
   - eslint 10→8/9：规则集变化，可能要修少量 lint 错误。
5. 验证：`npm run dev` 启动正常、`npm run build` 产物可用、页面手工点一遍核心流程。

**这是后续所有前端改动的前提**——若不先解决版本问题，3.2/3.3/3.8 的重构都建立在流沙上。

## 八、并行与关键路径

```
阶段一 P0 ──────────────────► (可立即上线 hotfix)
                              │
阶段二 版本核实 ──────────────►─┤
                              │
阶段三 后端重构 ──────────────►─┤   ← 关键路径（最耗时）
   3.1→3.2→3.3→{3.4,3.5,3.7,3.8,3.9 并行}→3.6
                              │
阶段四 前端重构 ──────────────►─┤
   4.1→4.2→4.3   4.4,4.5,4.6 并行
                              │
阶段五 工程化 ────────────────►─┤
   5.1(依赖三/四) 5.2(依赖四) 5.3 5.4(依赖三) 5.5
                              │
阶段六 收尾 ──────────────────► (持续)
```

**可并行**：阶段一三个 P0 任务互相独立；阶段三的 3.4/3.5/3.7/3.8 在 3.3 完成后可并行；前后端重构在各自依赖满足后可并行。

**关键路径**：阶段二 → 阶段三(3.1→3.2→3.3→3.5) → 阶段四(4.2) → 阶段五(5.1)。约 **10-12 人日**串行。
