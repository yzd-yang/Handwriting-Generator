# 00 · 项目优化方案总览

> 对象：`handwriting-web-main`（Vue 3 前端 + FastAPI 后端的手写文字生成 Web 应用）
> 日期：2026-06-20
> 用途：对项目做一次系统性的优化审查，产出可落地的修改/执行/验证方案。

## 一、这组文档是什么

本目录（`doc/re/`）是一套完整的项目优化交付物，按"发现问题 → 怎么改 → 怎么执行 → 怎么验证"四步组织，共 4 份文档：

| 文档 | 回答的问题 | 适合谁看 |
|------|------------|----------|
| **01-问题分析.md** | 项目有哪些可优化点？严重度如何？ | 所有人（先看这份） |
| **02-修改方案.md** | 每条问题具体怎么改？代码长什么样？ | 实施者（开发） |
| **03-执行方案.md** | 分几步做？先后顺序？工时？风险？ | 实施者、项目经理 |
| **04-验证方案.md** | 怎么证明改对了？怎么回归？怎么上线？ | 实施 + 测试 + 运维 |

## 二、核心结论（TL;DR）

### 共发现 29 个优化点

| 维度 | P0 严重 | P1 重要 | P2 一般 | P3 优化 | 小计 |
|------|---------|---------|---------|---------|------|
| 后端 | 3 | 8 | 2 | 1 | 14 |
| 前端 | 0 | 6 | 2 | 1 | 9 |
| 工程化 | 0 | 3 | 3 | 0 | 6 |
| **合计** | **3** | **17** | **7** | **2** | **29** |

### 必须立即处理的 3 个 P0

1. **🔴 安全：敏感信息硬编码** —— Sentry DSN、SECRET_KEY、GA/Clarity ID 直接写在源码里提交进 Git，需移到环境变量并在 Sentry 后台 rotate DSN。
2. **🔴 健壮性：`identify.py` 必现崩溃** —— 上传一张没有横线的图片就让 `/api/imagefileprocess` 返回 500（`cv2.HoughLinesP` 返回 None 未判空）。
3. **🔴 健壮性：BackgroundTask 吞错误** —— `generate_handwriting_impl` 内部的 429/500/400 错误响应被 `run_generation_task` 一律标成 `completed`，用户卡死在"生成中"。

### 本轮应处理的 17 个 P1（要点）

- **后端**：`app.py` 1683 行需拆分；导入即副作用无法单测；CPU 密集任务阻塞事件循环；临时文件清理逻辑过度复杂（180 行掩盖 PIL 句柄泄漏根因）；`requirements.txt` 与 `pyproject.toml` 版本分叉；CORS 全开 + 上传无校验；日志 DEBUG 级 + 无轮转。
- **前端**：⚠️ **`package.json` 依赖版本号严重异常**（vue-router 5.x / vite 8.x / ts 6.x / eslint 10.x / tailwind 4.x，这些稳定版尚不存在，**必须先核实是否被工具篡改**）；Vue CLI 残留 + Options/Composition 混用；`HomeView.vue` 500+ 行 40 字段平铺；错误处理薄弱多处静默吞异常；`v-html` XSS 风险；第三方脚本硬编码。
- **工程化**：无 CI 门禁；`release.config.js` 分支配错（`main` 而非 `master`）；前端无 Dockerfile（compose 引用不存在的镜像）；无强制 lint。

### 建议执行节奏（约 14-19 人日）

```
阶段一 P0 紧急修复（1-2 天）→ 可立即 hotfix 上线
阶段二 前端版本核实（0.5-1 天）→ 解锁所有前端工作
阶段三 后端重构（5-7 天）→ 关键路径
阶段四 前端重构（4-5 天）→ 可与阶段三部分并行
阶段五 工程化（3-4 天）→ CI/Docker/lint/单测
阶段六 收尾优化（按需）
```

## 三、如何使用这组文档

### 如果你是决策者（只看这份）

- 看"核心结论"，决定是否立项、排期。
- 重点决策：① 前端依赖版本是否要重置（高风险）；② 后端是否要大拆分（高工作量）；③ 是否引入 CI 强制门禁。

### 如果你是实施者

1. 通读 `01` 了解全貌。
2. 按 `03` 的阶段顺序开工，每接到一个任务：
   - 查 `02` 看具体改法和代码示例。
   - 改完按 `04` 第二节对应条目验收。
   - 合并前跑 `04` 第三节回归矩阵。
3. 高风险任务（后端拆分 3.3、前端拆分 4.2）务必 code review。

### 如果你是测试 / 运维

- `04` 是你的主战场：第三节回归矩阵、第五节上线清单、第六节上线后监控。
- 特别关注 Windows 验证（文件句柄问题）和 Sentry 错误率。

## 四、文档间的对应关系

四份文档**编号严格对齐**：

```
01-问题分析.md  2.3「BackgroundTask 吞错误」
      ↕ 对应同一问题
02-修改方案.md  2.3「BackgroundTask 吞错误 → 任务状态正确反映失败」
      ↕ 落成执行步骤
03-执行方案.md  阶段一·任务 2.3「错误传递修复」
      ↕ 验收标准
04-验证方案.md  2.3「错误传递修复（P0）」验收表
```

任何一份文档里看到 `2.3`、`3.4`、`4.1` 这样的编号，在其他三份里都能找到同一问题的不同视角。

## 五、几个关键提醒（实施前必读）

1. **前端版本号问题（3.1）是最大的不确定性**。`vue-router: ^5.1.0`、`vite: ^8.0.12`、`typescript: ~6.0.2` 这些版本号的稳定版**至今不存在**。必须先检查 `node_modules` 实际装的是什么，再决定是否全量重置依赖。这会直接影响所有前端改动的可行性。

2. **Windows 是主战场**。项目有 `pandoc-*-windows-x86_64.msi`、文件句柄删除问题（180 行重试代码）、`.bat` 清理脚本。所有涉及文件 IO 的改动（2.7 临时文件、2.8 线程池）**必须在 Windows 上验证**，Linux CI 通过不代表 Windows 能跑。

3. **后端拆分（2.4/3.3）是工作量与风险双高的任务**。`app.py` 1683 行混杂路由/业务/工具/全局副作用/死代码。建议严格按"先迁工具纯函数 → 再迁 service → 最后迁路由，每步一个 commit、每步手工测"的节奏，**不要一次性大改**。

4. **Sentry DSN 已泄漏到 Git 历史**。光改代码不够，必须在 Sentry 后台 **rotate DSN**，否则旧 DSN 仍可被滥用。

5. **不要在重构中混入新功能**。本轮是"治理"而非"扩展"，保持每次改动可独立 revert。

## 六、后续维护

- 本组文档是某个时间点的快照，代码演进后若与文档冲突，**以代码为准**并回头更新文档。
- 建议每完成一个阶段，在对应文档末尾追加"实际执行记录"（实际工时、遇到的问题、与方案的偏差）。
- 新发现的问题追加到 `01`，新方案追加到 `02`，保持编号体系。
