**位置**：`backend/app.py:303-312`、`backend/app.py:386`

```python
sentry_sdk.init(

SECRET_KEY = "437d75c5af744b76607fe862cf8a5a368519aca486d62c5fa69ba42c16809z88"
```

**问题**：
- Sentry DSN 直接硬编码在源码中并提交到 Git。

### 2.2 🔴 P0 健壮性：`identify.py` 存在多处必现崩溃

**位置**：`backend/identify.py:49-51`、`102-104`、`146-152`

```python
lines = cv2.HoughLinesP(...)
lines = sorted(lines, key=lambda x: x[0][1])  # lines 可能为 None
```

**问题**：

1. `cv2.HoughLinesP` 在检测不到直线时返回 `None`，直接 `sorted(None)` 抛 `TypeError`。
2. `lines_rotated` 为空时，`range(1, len(lines_rotated))` 不报错但 `avg_distance` 无值，后续 `round(avg_distance)` 抛 `NameError`/`UnboundLocalError`。
3. `np.where(row == 255)[0][0]` 当行无前景像素时索引越界（虽有 `.size != 0` 保护，但 left/right 计算路径仍有 `0` 兜底导致后续聚类全部为 0，`DBSCAN` 结果异常）。
4. 整个函数**无任何 try/except**，一次异常会让 `/api/imagefileprocess` 返回 500，且 `finally` 中临时文件已被覆盖逻辑错乱。

**触发条件**：用户上传一张**没有横线**的图片（如纯背景、照片、竖排文字）。

### 2.3 🔴 P0 健壮性：`generate_handwriting_impl` 在 BackgroundTask 内返回 Response 被丢弃

**位置**：`backend/app.py:766-812`、`run_generation_task` 内 `app.py:1198-1203`

```python
async def generate_handwriting_impl(...):
    ...
    if cpu_usage > 90:
        return JSONResponse({...}, status_code=429)   # ← 这个返回值
    if len(data["text"]) > 10000:
        return JSONResponse({...}, status_code=500)   # ← 和这个
    ...
```

**问题**：
- `generate_handwriting_impl` 内部多处 `return JSONResponse(...)`（CPU 过载 429、文本超长 500、缺字段 400、字体缺失 400 等）。
- 但它被 `run_generation_task` 在 `BackgroundTask` 中调用，调用方只读取 `response.body / response.status_code` 存入 task_store，**并标记 status="completed"**（见 `app.py:1209-1221`）。
- 结果：一个本该是"失败"的 429/500/400 响应，被当作 `completed` 存进队列，前端轮询到 `task_status=completed` 后去取 result，拿到的是一个 JSON 错误体却被当作 `application/json` 的成功结果——前端 `handleGenerationResultResponse` 按 content-type 分发，会**静默什么都不做**，用户卡死在"生成中"。

**根因**：函数职责混乱——既被同步路由复用（曾经），又被异步队列复用，但错误传递路径未统一。

### 2.7 🟠 P1 可靠性：临时文件清理逻辑过度复杂且有竞态

**位置**：`backend/app.py:52-231`（`safe_remove_directory` 等 4 个函数 + `cleanup_marked_directories`）

**问题**：
- `safe_remove_directory` 内部 `gc.collect()` + `time.sleep(0.2*attempt)` + 递归 + 兜底写 `.cleanup_later` 标记，逻辑极其复杂（~180 行），是典型的"用重试掩盖根因"。
- 根因其实是 **Windows 下 PIL 图片句柄未释放**（`app.py:175-194` 的 `safe_save_and_close_image` 把 `image.close()` 和 `gc.collect()` 都注释掉了！）。
- `safe_remove_single_file` 每次都 `os.chmod(file_path, 0o777)` —— **安全反模式**，会破坏文件权限模型。
- `tempfile.mktemp`（`pdf.py:37`）在 Python 3.12 已弃用且**有竞态条件**（TOCTOU），官方明确警告不要用。

### 2.8 🟠 P1 性能：CPU 密集任务在事件循环线程内执行，阻塞所有协程

**位置**：`backend/app.py:1019-1093`、`1135-1139`

```python
images = handwrite(text_to_generate, template)  # 惰性 map
for i, im in enumerate(images):                  # ← 这里消费，CPU 密集
    ...
    safe_save_and_close_image(im, image_path)    # PIL.save 是同步阻塞 IO
```

**问题**：
- `generate_handwriting_impl` 是 `async def`，但内部的 `handwrite()` 消费循环、PIL 保存、ZIP 打包、PDF 生成全是 **CPU/IO 密集的同步操作**，直接跑在事件循环线程上。
- 由于 `_generation_semaphore` 限并发为 2，实际不会把 CPU 打满，但**会阻塞同一个 worker 内的所有其他 async 协程**（包括 WebSocket 推送 `push_task_status_update`、轮询接口的 IO）。
- 代码注释（`app.py:22-25`）甚至解释了"为什么不放线程池"，但理由站不住：放 `run_in_executor` 才是 CPU 密集任务的正确姿势，能避免阻塞事件循环。
- gunicorn 只跑 `-w 1`（单 worker），事件循环一旦阻塞，全站响应变慢。

---

### 2.9 🟠 P1 并发：`asyncio.Semaphore` 在多 worker 下失效

**位置**：`backend/app.py:686-687`

```python
MAX_CONCURRENT_EXECUTIONS = 2
_generation_semaphore = asyncio.Semaphore(MAX_CONCURRENT_EXECUTIONS)
```

**问题**：
- 信号量是**进程内**的，gunicorn 若改成 `-w N`，每个 worker 各有独立信号量，真实并发 = `N × 2`，与设计意图不符。
- 同理 `task_websocket_connections`（`app.py:681`）是内存 dict，多 worker 下 WebSocket 连不到处理该任务的 worker，推送失效。
- 目前靠 `-w 1` 规避，但这与"用 Semaphore 控并发"的设计自相矛盾，且单 worker 无法利用多核。

---

### 2.10 🟠 P1 安全：CORS 完全放开 + 无文件类型校验

**位置**：`backend/app.py:343-349`

```python
app.add_middleware(CORSMiddleware, allow_origins=["*"], allow_credentials=True, ...)
```

**问题**：
- `allow_origins=["*"]` 与 `allow_credentials=True` 同时存在——按 CORS 规范这是**无效组合**（浏览器会拒绝带凭证的跨域请求），但语义上暴露了"想随便跨域"的意图。
- `/api/textfileprocess`、`/api/imagefileprocess` 用 `file.filename.endswith(".docx")` 判断类型，**完全信任客户端扩展名**，攻击者可上传任意内容的 `.docx`（如伪装的 zip bomb）。
- 上传无大小限制（`MAX_CONTENT_LENGTH` 已注释，`app.py:390`），可被大文件打满磁盘。

**位置**：`backend/app.py:321-340`

```python
logger.setLevel(logging.DEBUG)
ch.setLevel(logging.DEBUG)
fh.setLevel(logging.DEBUG)
fh = logging.FileHandler("logs/app.log")   # 无 RotatingFileHandler
```

## 三、前端问题（frontendVite/）

### 3.1 🟠 P1 工程：`vue-router` 版本号错误，存在重大隐患

**位置**：`frontendVite/package.json:45`

```json
"vue-router": "^5.1.0"
```

**问题**：
- Vue 3 对应的 vue-router 正式版是 **4.x**（`^4.x`）。vue-router 5.x 至今**并不存在正式稳定版**（`node_modules` 实际装的是 5.1.0，可能是某个 alpha/非官方发布）。
- `vuex: "^4.1.0"` 是对的（Vuex 4 对应 Vue 3），但项目里 Vuex 几乎没用（`store/index.js` 只有一个 `login_delete_message` 状态），建议直接去掉改用 Pinia 或组合式 API。
- 同理 `eslint: "^10.5.0"`、`vite: "^8.0.12"`、`typescript: "~6.0.2"` 等版本号都**远超当前真实稳定版**（vite 当前稳定版是 5.x/6.x，TypeScript 是 5.x），疑似被某个工具错误改写或来自未来时间线。这些"未来版本"要么装不上，要么装上是某个不规范发布。

> ⚠️ 这是最需要先和团队确认的点：这些版本号是真实可装的还是笔误/工具篡改？若是后者，整个 `package.json` 需要回归到现实可用版本。

### 3.2 🟠 P1 工程：Vue CLI 残留 + Options API 与 Composition API 混用

**位置**：`frontendVite/src/main.js`（存在）与 `frontendVite/src/main.ts`、`counter.ts`（Vite 模板残留，但实际已被 `package.json` 的 `type: module` + `main.js` 接管）

**问题**：

### 3.3 🟠 P1 可维护性：`HomeView.vue` 单文件 500+ 行，data() 40+ 字段

**位置**：`frontendVite/src/views/HomeView.vue:240-274`

**问题**：

### 3.4 🟠 P1 健壮性：`HomeView.vue` 错误处理薄弱，多处 `catch {}` 静默吞异常

**位置**：`frontendVite/src/views/HomeView.vue:313`、`453`

```js

...
```

**问题**：

### 4.2 🟠 P1 工程：根目录与各子包的 `package.json` 关系混乱

**位置**：根 `package.json`、`frontendVite/package.json`、`serverless/package.json`、`.mimocode/package.json`、`.opencode/`

**问题**：
- 根 `package.json` 只有 semantic-release，但 `package-lock.json`（227KB）在根——说明曾在根目录 `npm install` 过前端依赖，历史混乱。
- `.mimocode/`、`.opencode/`、`.workbuddy/` 是各种 AI 工具的工作目录，混在业务仓库里（部分已 gitignore，部分没有）。
- `format.py`（根目录）是个 sympy 实验脚本，与项目无关，是垃圾文件。

### 4.3 🟠 P1 工程：前端无 Dockerfile，docker-compose 引用了不存在的镜像

**位置**：`docker-compose.yml:3`（`image: frontend-handwriting:latest`）

**问题**：
- compose 文件直接用预构建镜像名，**没有前端 Dockerfile**，`docker compose up` 无法本地构建，只能靠外部 CI 推镜像。
- 后端 Dockerfile 在 `backend/dockerfile.backend`（非标准命名且在子目录），compose 也没用 `build:` 指向它，二者脱节。
- 没有 `.dockerignore`，构建上下文会把 `node_modules`、`.git`、字体文件全发过去。

### 4.4 🟡 P2 规范：无统一的代码格式化/lint 强制

**位置**：`frontendVite/.eslintrc.cjs`、`.prettierrc`（空）

**问题**：
- `.prettierrc` 是空文件（无配置）。
- ESLint 配置存在但 `npm run lint` 只在手动执行时跑，无 pre-commit hook（无 husky + lint-staged）。
- 后端**完全没有** lint（无 flake8/ruff/black 配置）。
- Python 代码缩进/引号风格不统一（混用单双引号、混用 f-string 和 % 格式化）。
