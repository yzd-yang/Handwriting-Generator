### 2.2 🔴 P0 `identify.py` 必现崩溃 → 防御性校验 + try/except

**改法**：对 OpenCV 检测结果全部判空，函数整体 try/except 返回安全默认值。

**代码示例**：

```python
# backend/identify.py
def identify_distance(filename):
    try:
        image = cv2.imread(filename)
        if image is None:
            raise ValueError("无法读取图片，文件损坏或格式不支持")

        gray = cv2.cvtColor(image, cv2.COLOR_BGR2GRAY)
        _, binary = cv2.threshold(gray, 150, 255, cv2.THRESH_BINARY_INV)
        kernel = np.ones((5, 5), np.uint8)
        dilated = cv2.dilate(binary, kernel, iterations=1)
        eroded = cv2.erode(dilated, kernel, iterations=1)
        edges = cv2.Canny(eroded, 50, 150, apertureSize=3)

        lines = cv2.HoughLinesP(edges, 1, np.pi / 180, 100,
                                minLineLength=100, maxLineGap=10)
        # 🟢 关键：检测不到直线时返回 None，必须判空
        if lines is None or len(lines) == 0:
            raise ValueError("图片中未检测到横线，无法识别边距/行距")

        lines = sorted(lines, key=lambda x: x[0][1])

        # ... 原 angles / rotated 逻辑不变，但加 lines_rotated 判空 ...
        lines_rotated = cv2.HoughLinesP(...)
        if lines_rotated is None or len(lines_rotated) == 0:
            lines_rotated = []

        # 计算 distances 时保证 len(lines_rotated) >= 2
        distances = []
        for i in range(1, len(lines_rotated)):
            d = lines_rotated[i][0][1] - lines_rotated[i-1][0][1]
            if d > 5:
                distances.append(d)

        # 🟢 所有可能为空的列表都先兜底，避免 get_avg_distance 崩
        safe = lambda lst, default=0: lst if lst else [default]
        avg_l = get_avg_distance(safe(l_whitespaces)) if l_whitespaces else 0
        avg_r = get_avg_distance(safe(r_whitespaces)) if r_whitespaces else 0
        avg_t = get_avg_distance(safe(t_whitespaces)) if t_whitespaces else 0
        avg_b = get_avg_distance(safe(b_whitespaces)) if b_whitespaces else 0
        avg_d = get_avg_distance(distances) if distances else 0

        return (round(avg_l), round(avg_r), round(avg_t), round(avg_b), round(avg_d))

    except Exception as e:
        print(f"identify_distance failed: {e}")
        # 🟢 返回全 0 兜底，让前端提示"无法识别"
        return 0, 0, 0, 0, 0
```

同时在 `app.py` 的 `/api/imagefileprocess` 路由里，若返回值全 0，应给前端一个明确提示（HTTP 200 + `{"warning": "..."}`，而不是 500）。

**替代方案**：把图片识别整体挪到线程池（见 2.8），并在失败时返回 422 + 错误消息，让前端引导用户换图。

### 2.3 🔴 P0 BackgroundTask 吞错误响应 → 任务状态正确反映失败

**根因**：`generate_handwriting_impl` 内部对业务错误（CPU 过载、文本超长、缺字段、字体缺失）用 `return JSONResponse(...)` 表达，但调用方 `run_generation_task` 不区分 `2xx` 与 `4xx/5xx`，一律标 `completed`。

**改法**：在 `run_generation_task` 里按 `response.status_code` 分流——非 2xx 标 `failed`。

**代码示例**：

```python
# backend/app.py: run_generation_task
async def run_generation_task(task_id, base_url, payload):
    set_generation_task(task_id, status="processing", stage="started", ...)
    await push_task_status_update(task_id)
    try:
        def progress_notify(**kwargs):
            set_generation_task(task_id, **kwargs)
            asyncio.create_task(push_task_status_update(task_id))

        async with _generation_semaphore:
            set_generation_task(task_id, stage="executing", message="正在生成中")
            await push_task_status_update(task_id)
            response = await generate_handwriting_impl(
                base_url=base_url, progress_hook=progress_notify, **payload
            )

        status_code = response.status_code
        # 🟢 关键：非 2xx 一律视为失败
        if not (200 <= status_code < 300):
            # 解析错误体里的 message
            try:
                body_text = response.body.decode("utf-8") if response.body else ""
                err_msg = json.loads(body_text).get("message", body_text)
            except Exception:
                err_msg = f"生成失败 (HTTP {status_code})"
            set_generation_task(
                task_id, status="failed", error_message=err_msg,
                stage="failed", message="任务处理失败",
            )
            await push_task_status_update(task_id)
            return  # 🟢 不写结果文件，不标 completed

        # 成功路径：原逻辑
        set_generation_task(
            task_id, status="completed",
            response_status_code=status_code,
            response_content_type=response.headers.get("content-type") or response.media_type,
            response_body=response.body,   # 由 task_store 写盘
            stage="completed", message="任务处理完成", progress=100,
        )
        await push_task_status_update(task_id)
    except Exception as e:
        logger.exception("Generation task failed, task_id=%s", task_id)
        set_generation_task(task_id, status="failed", error_message=str(e),
                            stage="failed", message="任务处理失败")
        await push_task_status_update(task_id)
```

**配套**：前端 `HomeView.vue` 的 `handleGenerationResultResponse` 按 `task_status === 'failed'` 分支，读 `error_message` 用 Swal 提示用户，不要按 content-type 静默分发。

**更彻底的替代方案**（🟡，工作量大）：让 `generate_handwriting_impl` 不再 `return Response`，而是**抛异常**（自定义 `GenerateError(status_code, message)`），由 `run_generation_task` 捕获并落 `failed`。这样函数职责单一（只产出成功的二进制内容或抛错），不再混用两种返回约定。推荐中期重构时采用。

### 2.7 🟠 P1 临时文件清理过度复杂 → 修根因 + 简化

**根因**：Windows 下 PIL 图片对象未释放文件句柄，导致删除报 `PermissionError`。`safe_save_and_close_image`（`app.py:175-194`）把 `image.close()` 和 `gc.collect()` 都注释掉了，于是后续用 180 行重试逻辑兜底。

**改法**：

1. **恢复 PIL 句柄释放**：保存后 `image.close()`，必要时 `gc.collect()`。
2. **简化删除函数**：标准库 `shutil.rmtree(path, ignore_errors=True)` 足够，删掉 180 行重试。
3. **去掉 `os.chmod(0o777)`**：安全反模式，不应改文件权限。
4. **`pdf.py` 弃用 `tempfile.mktemp`** → 用 `tempfile.NamedTemporaryFile(delete=False)`。

**代码示例**：

```python
# backend/services/cleanup.py（重构后，<30 行）
import gc
import shutil
from pathlib import Path

def safe_save_and_close_image(image, image_path):
    """保存并立即释放 PIL 句柄，避免 Windows 下文件锁定。"""
    try:
        image.save(image_path)
        image.close()          # 🟢 恢复：释放句柄
        return True
    except Exception as e:
        logger.error("Failed to save image %s: %s", image_path, e)
        return False

def remove_dir(path):
    """删除目录，失败仅记日志不重试。残留由定时清理兜底。"""
    try:
        p = Path(path)
        if p.exists():
            shutil.rmtree(p, ignore_errors=True)
            gc.collect()
    except Exception as e:
        logger.warning("remove_dir failed %s: %s", path, e)

def remove_file(path):
    try:
        Path(path).unlink(missing_ok=True)
    except Exception as e:
        logger.warning("remove_file failed %s: %s", path, e)
```

```python
# backend/utils/pdf_build.py（修复 mktemp）
def generate_pdf(images):
    temp_dir = tempfile.mkdtemp(dir="./temp")
    try:
        pdf_document = fitz.open()
        for i, img in enumerate(images):
            temp_img_path = os.path.join(temp_dir, f"image{i}.jpg")
            img.save(temp_img_path, format="JPEG", quality=95)
            img.close()                          # 🟢 释放句柄
            w, h = img.size  # 注意：close 后仍可读 .size（PIL 行为）
            page = pdf_document.new_page(width=w, height=h)
            page.insert_image(fitz.Rect(0, 0, w, h), filename=temp_img_path)
        # 🟢 用 NamedTemporaryFile 替代弃用的 mktemp
        with tempfile.NamedTemporaryFile(suffix=".pdf", dir="./temp", delete=False) as f:
            temp_pdf_path = f.name
        pdf_document.save(temp_pdf_path)
        pdf_document.close()
        return temp_pdf_path
    finally:
        shutil.rmtree(temp_dir, ignore_errors=True)
```

> **注意**：`img.close()` 后 `img.size` 在 PIL 中仍可读（size 是已解析属性），但保险起见可在 close 前先记录 `w, h = img.size`。

**替代方案**（更激进）：完全不落地中间图片，handwrite 的结果直接 `fitz.Pixmap` / PIL 内存对象塞进 PDF / ZIP（`zipfile.ZipFile.writestr` 支持写 bytes）。可省掉整个临时目录，根治文件句柄问题。工作量较大，作为后续优化。

### 2.8 🟠 P1 CPU 密集阻塞事件循环 → 线程池

**改法**：把 `generate_handwriting_impl` 里 `handwrite()` 消费循环、PIL 保存、ZIP 打包、PDF 生成等同步阻塞部分，用 `asyncio.to_thread()`（或 `run_in_executor`）包起来，让出事件循环。

**代码示例**：

```python
# backend/services/handwriting.py
import asyncio

def _render_and_pack_sync(template, text, is_preview, full_preview, base_url, data):
    """同步函数：所有 CPU/IO 密集操作都在这里，跑在线程池。"""
    images = handwrite(text, template)
    # ... 原 for 循环 + PIL.save + ZIP + PDF 逻辑 ...
    return result_bytes, content_type, headers

async def generate_handwriting_impl(...):
    # 参数校验、CPU 检查等轻量逻辑保持 async
    ...
    # 🟢 重活挪到线程池
    result_bytes, content_type, headers = await asyncio.to_thread(
        _render_and_pack_sync, template, text_to_generate, is_preview, full_preview, base_url, data
    )
    return Response(content=result_bytes, media_type=content_type, headers=headers)
```

**进度上报**：`progress_hook` 是 async 的（要触发 WebSocket 推送），在线程里不能直接 await。用 `asyncio.run_coroutine_threadsafe` 把进度回传到事件循环：

```python
def _render_and_pack_sync(..., loop, progress_async):
    # progress_async 是一个线程安全的回调，内部用 run_coroutine_threadsafe
    progress_async("rendering", "正在处理第 1 页", 60)
```

**简化方案**（🟡，工作量小）：如果不想拆函数，至少把**最耗时的 `handwrite` 消费循环**单独 `await asyncio.to_thread(...)`，其余保留。先解决"渲染时 WebSocket 推送卡顿"这个用户可感知问题。

### 2.9 🟠 P1 多 worker 下 Semaphore/WebSocket 失效 → 单 worker 前提下文档化，或改用 Redis

**现状**：gunicorn `-w 1`，单 worker，问题暂时不会暴露。

**改法 A（🟢 推荐，低成本）**：明确约束"必须单 worker"，在 README / dockerfile 注释里写清楚，并把 `_generation_semaphore` 的注释说明这是进程内限流。

**改法 B（🟡，高成本）**：要支持多 worker，需引入外部协调：
- 并发计数：用 Redis（`INCR`/`DECR` + TTL）替代 `asyncio.Semaphore`。
- WebSocket 推送：用 Redis Pub/Sub，每个 worker 订阅自己持有的 task_id 频道，任意 worker 完成任务后 publish。
- 任务状态：`task_store` 从 SQLite 改 Redis（或保留 SQLite 但加 advisory lock）。

**推荐**：当前负载（RPS ~0.5，Semaphore 2）单 worker 完全够用，**采用方案 A**，把"单 worker"作为部署约束写入运维文档。多核利用靠"开多个容器实例 + nginx 负载均衡"，而非单进程多 worker。

### 2.14 🔵 P3 queue 查询优化 → 单条 SQL

**改法**：把 4 次查询合并为 1 条带条件的聚合。

```python
# backend/task_store.py
def get_queue_metrics(task_id: str) -> dict[str, int]:
    conn = _connect()
    try:
        row = conn.execute(
            "SELECT status, created_at FROM generation_tasks WHERE task_id = ?",
            (task_id,),
        ).fetchone()
        if row is None:
            return {"queue_pending_count": 0, "queue_ahead_count": 0,
                    "processing_count": 0, "active_task_count": 0}

        agg = conn.execute("""
            SELECT
                SUM(status = 'pending')    AS pending,
                SUM(status = 'processing') AS processing
            FROM generation_tasks
        """).fetchone()
        pending = agg["pending"] or 0
        processing = agg["processing"] or 0

        ahead = 0
        if row["status"] == "pending":
            created = row["created_at"] or 0
            ahead = conn.execute("""
                SELECT COUNT(*) FROM generation_tasks
                WHERE status = 'pending'
                  AND (created_at < ? OR (created_at = ? AND task_id < ?))
            """, (created, created, task_id)).fetchone()[0]

        return {
            "queue_pending_count": pending,
            "queue_ahead_count": ahead,
            "processing_count": processing,
            "active_task_count": pending + processing,
        }
    finally:
        conn.close()
```

从 4 条 SQL 降到 2 条（聚合 + ahead）。若要更激进，可加 30 秒内存缓存（`functools.lru_cache` 带过期）。

```js
// src/composables/useFormParams.js
import { reactive, watch } from 'vue'

const STORAGE_KEY = 'handwrite:form-params'
const DEFAULTS = {
  text: '', font_size: '120', line_spacing: '180',
  left_margin: '80', top_margin: '80', right_margin: '80', bottom_margin: '80',
  // ... 全部字段 ...
}

export function useFormParams() {
  const params = reactive(loadFromStorage())

  function loadFromStorage() {
    try {
      const saved = JSON.parse(localStorage.getItem(STORAGE_KEY) || '{}')
      return { ...DEFAULTS, ...saved }
    } catch {
      return { ...DEFAULTS }
    }
  }

  watch(params, (val) => {
    localStorage.setItem(STORAGE_KEY, JSON.stringify(val))
  }, { deep: true })

  function reset() {
    Object.assign(params, DEFAULTS)
  }

  function buildFormData() {
    const fd = new FormData()
    for (const [k, v] of Object.entries(params)) {
      fd.append(k, v)
    }
    return fd
  }

  return { params, reset, buildFormData }
}
```

> 关键改进：用 `Object.entries` 自动 append，**字段增删只改 `DEFAULTS`**，不再 30 行手写 `formData.append`。
